Spartan IT · arquitectura · v1.0

El MCP no muere,
se envuelve.

Cuatro fases en la madurez de un agente con LLM: desde un MCP server que vos consultás con Claude Desktop, hasta un producto SaaS multi-tenant. Para entender en cuál estás, cuándo saltar a la siguiente, y qué del trabajo ya hecho se mantiene en cada salto.

tesis · el trabajo del MCP es el core técnico que sobrevive a todas las fases. lo que cambia entre una y otra es quién es el usuario, quién paga el modelo, y qué capa nueva se agrega encima.

nota · el modelo mental aplica igual con Anthropic, OpenAI, Gemini o modelos locales (Ollama, vLLM). cambia el proveedor del LLM, no la estructura de las fases.
no confundir · las "fases 00-03" acá refieren al producto entero (mcp+claude desktop → agente propio → saas multi-tenant). la app de dimensiones usa los mismos nombres pero aplica las fases a cada dimensión por separado (auth, rag, compliance, etc, cada una con su propio ritmo). son modelos complementarios, no el mismo.

› arquitectura · qué controla cada parte

Fase 00 — Claude.ai puro Explorar

Vos abrís claude.ai (o la app), tipeás preguntas, recibís respuestas. Sin tools, sin conexión a tus datos, sin nada custom. La fase que muchos saltean sin darse cuenta porque ya tienen una cuenta.

Sirve para

  • Validar si la idea de un agente para tu cliente tiene sentido conceptual antes de gastar tiempo en código.
  • Tareas personales de productividad — escribir, resumir, traducir.
  • Calibrar qué tan bueno es el modelo para tu dominio sin ninguna infraestructura.

No sirve para

  • Acceder a datos privados del cliente (bases SQL, FTP, archivos internos).
  • Construir cualquier producto reproducible.
  • Automatizar workflows.

Fase 01 — MCP + Claude Desktop Conectar

Armás un MCP server (en tu GCP, con tu Laravel o lo que sea) que expone funciones para consultar las bases del cliente. Lo conectás a Claude Desktop. Vos / Alberto / Tomás se conectan al cliente "hablando con Claude", y Claude usa las tools del MCP para leer datos reales. Donde están hoy con el lab.

Sirve para

  • Demos rápidas a clientes (les mostrás lo que se puede hacer con sus datos en una pantalla).
  • Exploración interna del equipo — Alberto se conecta a la base del lab y prueba consultas reales.
  • Validar qué consultas tienen sentido antes de invertir en una UI.
  • Debugging del lado tools (la lógica SQL, el manejo de errores, los formatos de salida).

También puede servir como producto final si

  • El cliente tiene equipo técnico interno cómodo configurando Claude Desktop.
  • El presupuesto es ajustado y no justifica desarrollar UI propia.
  • El alcance es chico (3-5 usuarios) y el cliente ya paga su propia suscripción de Claude.
  • No necesitás cobrar por uso del modelo, solo por el setup del MCP.

No sirve para

  • Que un usuario sin equipo técnico (como Luciana) lo use directamente — necesitaría instalar Claude Desktop, configurar el MCP server, autenticar.
  • Controlar el system prompt — heredás el de Claude Desktop (10-15k tokens, no podés tocarlo).
  • Optimizar costos de tokens — el overhead es fijo y enorme.
  • Personalizar la UI o integrar con un workflow propio del cliente.

El malentendido común

Mucha gente arma el MCP server pensando que es "el producto". En la mayoría de los casos no lo es. Es la herramienta interna del equipo técnico para construir y validar. El producto cliente-facing vive en la fase 02. La excepción son los casos del recuadro de arriba.

Fase 02 — Agente propio (cliente-facing) Operar

Armás una web propia donde el cliente del cliente entra y habla con el agente. El backend (Laravel) define el system prompt, gestiona el historial, ejecuta las tools y llama directo al API de Anthropic. Las funciones que escribiste en el MCP server siguen siendo las mismas: solo cambia que ahora las invoca tu código en vez del protocolo MCP.

Lo que aparece nuevo

  • System prompt propio, versionado, editable sin redeploy.
  • Control total de las llamadas al API — podés aplicar prompt caching, semantic cache, todo lo que vimos en la otra app.
  • UI a medida — colores del cliente, integraciones con su workflow, autenticación propia.
  • Facturación propia — el cliente paga un fee mensual a Spartan, vos pagás el API.
  • Autenticación, roles y permisos por usuario.
  • Logs de auditoría: quién consultó qué, cuándo, con qué resultado. Crítico para clientes regulados (lab, salud).
  • Telemetría detallada de uso, errores, latencia.

Cuándo conviene saltar

  • El cliente del lab dice "esto está bueno, lo quiero usar con mi equipo".
  • Querés cobrar por el servicio (no se puede facturar uso de Claude Desktop a un tercero).
  • Necesitás integrar con un workflow del cliente (por ejemplo, abrir un ticket desde el chat).
  • Estás viendo gastos altos en tokens y querés aplicar prompt caching / semantic cache.
  • El cliente está regulado y necesita trazabilidad de cada consulta.

Nota técnica avanzada

El MCP server también se puede usar desde el agente Laravel — Anthropic tiene un cliente MCP que podés llamar desde código. En la práctica, para Spartan, conviene reescribir las tools como clases PHP nativas en fase 02 porque te da más control. El MCP server del lab sigue vivo en paralelo para uso interno.

Fase 03 — SaaS multi-tenant Escalar

Mismo agente core, múltiples clientes con configuración propia. Spartan se convierte en plataforma: un código base, distintos system prompts por cliente, datos aislados, billing por consumo. Es la fase donde dejás de ser una consultora y pasás a ser un producto.

Lo que aparece nuevo

  • Tabla tenants + middleware de routing por cliente en cada request.
  • Configuración por cliente: prompt, tools habilitadas, modelo, quotas, límites.
  • Aislamiento de datos — cada cliente tiene su conexión a base, sus credenciales, su namespace.
  • Métricas agregadas para el dashboard interno de Spartan (uso por cliente, ingresos, costos).
  • Onboarding de clientes nuevos sin tocar código — todo se hace por config.

Cuándo conviene saltar

  • Tenés 2-3 clientes con un agente que es "parecido pero distinto". Si no, fase 02 con configs estáticas alcanza.
  • Empezás a perder tiempo en mantener varios deploys casi iguales.
  • Querés ofrecer un self-service donde un cliente nuevo se da de alta solo (futuro, no urgente).

El riesgo

Es fácil saltar a fase 03 antes de tiempo y construir abstracciones para clientes que todavía no tenés. Regla práctica: no abstraigas hasta tener el tercer cliente. Hasta entonces, dos instancias de fase 02 con configs estáticas son más rápidas y menos riesgosas.

La misma función en cada fase

Para volver visible la tesis: el core del MCP no se tira. Tomamos una función concreta del lab y la mostramos cómo se ve en cada fase. Lo que aparece en color es lo que cambia respecto a la fase anterior.

El escenario

El usuario pregunta: "¿cuántos pacientes mayores de 60 tienen análisis pendientes?" El agente necesita una función que reciba edad mínima y devuelva el conteo. Es la misma necesidad en las cuatro fases — sólo cambia quién la invoca y cómo.

› función a implementar: contar_pacientes_con_analisis_pendientes(edad_minima: int) → int
FASE 00 Claude.ai puro
no aplica
no hay función En esta fase no hay forma de conectar a la base. Lo más que podés hacer es pegar el resultado a mano después de correr la query en tu cliente SQL. Por eso fase 00 no escala más allá de exploraciones puntuales.

SELECT COUNT(*)
FROM paciente p
JOIN orden_analisis oa ON oa.paciente_id = p.id
WHERE p.edad >= 60
  AND oa.estado = 'pendiente';
FASE 01 Tool del MCP server
Python · MCP SDK
nuevo · aparece la función Acá vive el core que va a sobrevivir en todas las fases siguientes. La definís una vez, la registrás como tool MCP, y Claude Desktop la descubre por el protocolo. El cuerpo SQL es el mismo que en fase 00, pero envuelto como función. 
# server.py — MCP server
from mcp.server.fastmcp import FastMCP
import psycopg2

mcp = FastMCP("laboratorio")

@mcp.tool()
def contar_pacientes_con_analisis_pendientes(edad_minima: int) -> int:
    """Devuelve la cantidad de pacientes mayores o iguales a la edad
    indicada que tienen al menos una orden de análisis pendiente."""
    conn = psycopg2.connect(DB_URL)
    with conn.cursor() as cur:
        cur.execute("""
            SELECT COUNT(*)
            FROM paciente p
            JOIN orden_analisis oa ON oa.paciente_id = p.id
            WHERE p.edad >= %s
              AND oa.estado = 'pendiente'
        """, (edad_minima,))
        return cur.fetchone()[0]

if __name__ == "__main__":
    mcp.run()
FASE 02 Tool como clase del agente Laravel
PHP · Laravel
cambio · misma lógica, otro envoltorio El cuerpo SQL es literalmente el mismo. Lo que cambia: ahora la función vive como clase Tool dentro de tu Laravel, el agente la conoce por su nombre, y los parámetros se validan con un schema que va al system prompt como definición de tool. No hay más MCP en el medio — el agente la invoca como método PHP. 
// app/Tools/ContarPacientesConAnalisisPendientesTool.php
namespace App\Tools;

use Illuminate\Support\Facades\DB;

class ContarPacientesConAnalisisPendientesTool
{
    public static string $name = 'contar_pacientes_con_analisis_pendientes';

    public static function schema(): array
    {
        return [
            'name' => self::$name,
            'description' => 'Devuelve la cantidad de pacientes mayores '
                . 'o iguales a la edad indicada que tienen al menos '
                . 'una orden de análisis pendiente.',
            'input_schema' => [
                'type' => 'object',
                'properties' => [
                    'edad_minima' => [
                        'type' => 'integer',
                        'description' => 'Edad mínima en años, inclusive.',
                    ],
                ],
                'required' => ['edad_minima'],
            ],
        ];
    }

    public function execute(array $input): int
    {
        // el SQL es el mismo de fase 01 — solo cambia el adaptador
        return DB::connection('laboratorio')
            ->selectOne("
                SELECT COUNT(*) AS total
                FROM paciente p
                JOIN orden_analisis oa ON oa.paciente_id = p.id
                WHERE p.edad >= ?
                  AND oa.estado = 'pendiente'
            ", [$input['edad_minima']])->total;
    }
}
FASE 03 Tool multi-tenant
PHP · Laravel + tenant context
cambio · resolución dinámica de tenant El SQL sigue siendo el mismo. La clase es casi idéntica a fase 02. Lo único que aparece es que la conexión a base se resuelve dinámicamente según qué cliente está usando la plataforma en este request. Una línea cambia. El resto, intacto. 
// app/Tools/ContarPacientesConAnalisisPendientesTool.php
namespace App\Tools;

use App\Services\TenantContext;
use Illuminate\Support\Facades\DB;

class ContarPacientesConAnalisisPendientesTool
{
    public static string $name = 'contar_pacientes_con_analisis_pendientes';

    public function __construct(private TenantContext $tenant) {}

    public static function schema(): array
    {
        // idéntico a fase 02
        return [ /* ... mismo schema ... */ ];
    }

    public function execute(array $input): int
    {
        // la única línea que cambia: la conexión sale del tenant activo
        $conn = $this->tenant->databaseConnection();

        return DB::connection($conn)
            ->selectOne("
                SELECT COUNT(*) AS total
                FROM paciente p
                JOIN orden_analisis oa ON oa.paciente_id = p.id
                WHERE p.edad >= ?
                  AND oa.estado = 'pendiente'
            ", [$input['edad_minima']])->total;
    }
}
› la moraleja

El SQL es el activo. Todo lo demás es envoltura.

Mirando las cuatro versiones lado a lado, el SELECT COUNT(*) aparece idéntico en las cuatro fases. Lo que cambia es el envoltorio: primero ejecutado a mano, después como tool MCP, después como clase PHP, después con resolución de tenant. El trabajo difícil — entender el modelo de datos del laboratorio, escribir queries correctas, manejar edge cases — ese trabajo se hace una vez en fase 01 y se reusa en todas las fases superiores. Por eso saltar de fase no es empezar de cero: es agregar una capa.
